Interlude IV

Async Internals

A surgical dissection of futures, wakers, and the async state machine

"Async/await is not magic‚Äîit is a state machine. Understanding the state machine is understanding async Rust."

The async and await keywords hide a sophisticated machinery. When you write an async function, the compiler transforms it into a state machine‚Äîa struct that implements the Future trait. This interlude dissects that machinery.

The Future Trait

At its core, async Rust is built on a simple trait:

pub trait Future {
    type Output;
    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) 
        -> Poll<Self::Output>;
}

A Future represents a value that may not be ready yet. The poll method checks if the value is ready:

  • Returns Poll::Ready(T) if the value is available
  • Returns Poll::Pending if the value is not yet available

Async as State Machine

Consider this async function:

async fn fetch_data() -> String {
    println!("Starting");
    let response = make_request().await;
    println!("Got response");
    let data = parse(response).await;
    println!("Done");
    data
}

The compiler transforms this into an enum representing the states:

Async State Machine: enum FetchDataFuture { Start, WaitingForRequest { request: RequestFuture }, WaitingForParse { parse: ParseFuture }, Done, } impl Future for FetchDataFuture { fn poll(mut self: Pin<&mut Self>, cx: &mut Context) -> Poll<String> { loop { match *self { Start => { println!("Starting"); let request = make_request(); *self = WaitingForRequest { request }; } WaitingForRequest { ref mut request } => { match request.poll(cx) { Poll::Ready(response) => { println!("Got response"); let parse = parse(response); *self = WaitingForParse { parse }; } Poll::Pending => return Poll::Pending, } } // ... more states } } } }

Async futures are often self-referential‚Äîthe state machine contains pointers to its own data. Pin guarantees that the future will not move in memory, making these self-references valid. Without Pin, the async machinery would be unsound.

The Waker

When a future returns Poll::Pending, it must arrange to be polled again later. This is done through the Waker:

fn poll(self: Pin<&mut Self>, cx: &mut Context) -> Poll<Self::Output> {
    if self.is_ready() {
        Poll::Ready(self.value)
    } else {
        // Register waker to be called when ready
        self.register_waker(cx.waker());
        Poll::Pending
    }
}

The waker is a callback that the runtime will invoke when the future should be polled again. This enables efficient async I/O without blocking threads.

The Runtime

Async Rust requires a runtime to execute futures. Popular choices include:

  • Tokio ‚Äî Full-featured async runtime for production
  • async-std ‚Äî Standard library-like async primitives
  • smol ‚Äî Small, portable async runtime

The runtime provides:

  • A reactor for async I/O events
  • An executor to poll futures
  • Timers and other async primitives
