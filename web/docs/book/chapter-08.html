The Trickster: Closures and Dynamic Dispatch

The shape-shifter who captures environment, bends rules, and dances between static and dynamic worlds

"The trickster is a boundary-crosser. He breaks through the boundaries of social reality, challenges the established order, and reveals the ambiguity and uncertainty that underlie all systems."

‚Äî Lewis Hyde

There exists in Rust a construct that defies categorization. It is a function, yet it carries data. It has a type, yet that type is often hidden. It can be stored, passed, and called‚Äîyet its exact nature is known only to the compiler. This is the closure‚Äîthe Trickster of Rust.

Closures capture their environment. They sneak variables from surrounding scope into their own body, creating a self-contained unit of behavior and state. They are the shapeshifters of the type system‚Äîsometimes acting as function pointers, sometimes as unique anonymous structs, always adapting to their context.

üÉè The Trickster Speaks

"I am the closure. I capture what I need from the world around me‚Äîby reference, by mutable reference, or by ownership. I am efficient when I can be, extravagant when I must. The compiler deduces my nature from how I use my captures. I am never quite what I seem."

"Store me in a Box. Pass me to a thread. Call me when you need me. I adapt. I survive. I trick the borrow checker into letting me do what must be done."

Closure Syntax and Capture

Closures in Rust use the syntax |params| body:

let factor = 2;

// Closure that captures 'factor' by reference
let multiply = |x| x * factor;

println!("{}", multiply(5));  // Prints 10

// The closure 'multiply' holds a reference to 'factor'
// Its type is inferred as impl Fn(i32) -> i32

The compiler determines how the closure captures variables:

  • &T (immutable reference): When the closure only reads the variable
  • &mut T (mutable reference): When the closure modifies the variable
  • T (by value/ownership): When the closure moves the variable (e.g., when returning the closure from a function)

The Three Fn Traits

Closures implement one or more of these traits:

// Fn: Can be called multiple times, only immutable borrow
fn call_fn(f: impl Fn() -> i32) -> i32 {
    f() + f()  // Called twice
}

// FnMut: Can be called multiple times, may mutate state
fn call_fn_mut(mut f: impl FnMut()) {
    f();  // Each call may modify captured state
    f();
}

// FnOnce: Consumes self on call, can only be called once
fn call_fn_once(f: impl FnOnce() -> String) -> String {
    f()  // After this, f is consumed
}

"Closures are not function pointers‚Äîthey are structs containing captured state. When you pass a closure, you are passing data. The compiler monomorphizes the call, inlining when possible. Zero-cost abstraction applies here too."

Dynamic Dispatch with Trait Objects

Sometimes you need to store different closures in the same variable. This requires dynamic dispatch via trait objects:

// Box can hold any closure that implements Fn()
let handlers: Vec<Box<dyn Fn()>> = vec![
    Box::new(|| println!("First")),
    Box::new(|| println!("Second")),
];

for handler in handlers {
    handler();  // Dynamic dispatch through vtable
}

The tradeoff is clear:

  • Static dispatch (impl Trait, generics): Fast, monomorphized, compile-time determined
  • Dynamic dispatch (dyn Trait): Flexible, runtime polymorphism, slight overhead

The Move Keyword

Sometimes you need to force ownership capture, especially when sending closures to threads:

let data = vec![1, 2, 3];

// Without 'move', the closure would try to borrow 'data'
// But the thread might outlive the current scope!

std::thread::spawn(move || {
    // 'data' is moved into the closure, then into the thread
    println!("{:?}", data);
}).join().unwrap();

// 'data' is no longer available here‚Äîit was moved
