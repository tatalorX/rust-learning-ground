A surgical dissection of dynamic memory allocation at the system level

"The heap is the wilderness of memoryâ€”unstructured, unbounded, requiring constant vigilance. Unlike the orderly stack, the heap has no automatic cleanup. Every allocation is a commitment; every deallocation, a responsibility fulfilled."

In the first interlude, we dissected the stack frameâ€”a structured, predictable region of memory governed by the strict discipline of function calls and returns. Now we turn our attention to the heap: the dynamic memory region where persistence is purchased at the price of vigilance.

The heap is the operating system's gift to programs that need memory with a lifespan beyond a single function call. But this gift is not without strings attached. The programâ€”not the compiler, not the runtimeâ€”must track what has been allocated and ensure it is returned when no longer needed.

The Heap Block Structure

When you allocate memory on the heap using Box::new, Vec::new, or String::new, the allocator does not simply hand you raw bytes. It constructs a structureâ€”a block with metadata preceding the data you requested.

Consider what happens when you execute:

let patient = Box::new(Patient { id: 42, name: "Smith".to_string() });

The heap allocator performs several operations atomically:

  1. Calculates total size: metadata header + data alignment padding + actual data
  2. Searches the free list for a sufficiently large block
  3. If no block exists, requests more memory from the OS via brk/mmap
  4. Writes allocation metadata (size, flags) to the block header
  5. Returns a pointer to the user-accessible region (after the header)

Rust's Box: The Smart Pointer

Rust's Box<T> is not merely a raw pointer. It is a smart pointerâ€”a type that encapsulates both the pointer and the responsibility for cleanup.

// The Box struct (simplified)
pub struct Box<T: ?Sized> {
    ptr: Unique<T>,  // Non-null, owning pointer to heap memory
}

// When Box goes out of scope...
impl<T> Drop for Box<T> {
    fn drop(&mut self) {
        // 1. Drop the contents (recursively for nested types)
        // 2. Deallocate the memory block
        unsafe {
            self.ptr.drop_in_place();  // Call T's destructor
            dealloc(self.ptr.as_ptr(), Layout::new::<T>());
        }
    }
}

This is the genius of Rust's ownership system applied to heap memory: the compiler inserts the destructor call automatically, and the destructor ensures proper cleanup. The programmer cannot forget to free the memoryâ€”the type system guarantees it.

The Global Allocator Interface

Rust allows you to replace the global allocator. This is essential for embedded systems, game engines, and any application with specific memory requirements.

// Using jemalloc for better fragmentation behavior
use jemallocator::Jemalloc;

#[global_allocator]
static GLOBAL: Jemalloc = Jemalloc;

// Now all Box::new, Vec::new, String::new use jemalloc
ğŸ”¬ Surgical Precision: Allocator Selection

Choosing an allocator is like choosing a surgical approach:

  • System allocator: Good general-purpose, uses malloc/free
  • jemalloc: Excellent for concurrent workloads, reduces fragmentation
  • mimalloc: Small memory footprint, fast for small allocations
  • Bump allocator: Ultra-fast, arena-style, no individual deallocation
  • Pool allocator: Fixed-size blocks, deterministic, no fragmentation
