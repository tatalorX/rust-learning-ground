Handling Text Streams

Rust makes working with text streams ergonomic through the std::io traits:

use std::io::{self, BufRead, Write};

// Read from stdin, transform, write to stdout
let stdin = io::stdin();
let stdout = io::stdout();

for line in stdin.lock().lines() {
    let line = line?;
    let transformed = line.to_uppercase();
    writeln!(stdout.lock(), "{}", transformed)?;
}

This program follows the Unix philosophy: it reads text, transforms it, outputs text. It can be composed with other tools using pipes.

CLI Application Structure

A well-structured Rust CLI application separates concerns:

// src/main.rs - Entry point, argument parsing
// src/lib.rs  - Library code, testable
// src/cli.rs  - Command-line interface
// src/core.rs - Business logic

This structure enables testing, reuse, and clear separation of concerns.

Exit Codes

Unix programs communicate success and failure through exit codes:

use std::process;

fn main() {
    if let Err(e) = run() {
        eprintln!("Error: {}", e);
        process::exit(1);  // Failure
    }
    // Implicit exit 0 - Success
}

Following these conventions makes your Rust programs good Unix citizens.

üî¨ Surgical Precision: Process Spawning

When you spawn a process, the kernel must:

  1. Allocate a new PID and process table entry
  2. Clone the parent's memory space (copy-on-write)
  3. Duplicate file descriptors
  4. Set up the new process's stack and heap
  5. Schedule the new process

This is why process spawning is expensive compared to threads, and threads are expensive compared to async tasks.
