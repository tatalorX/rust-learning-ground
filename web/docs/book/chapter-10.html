Chapter Ten

Zero-Cost Abstractions

What you do not use, you do not pay for‚Äîand what you do use, you could not hand-code any better

"C++ is a language where you pay for what you use. Rust is a language where you only pay for what you use‚Äîand you cannot hand-code better than what the compiler produces."

‚Äî Rust Design Principle

üé≠ The Optimizer Speaks

"I am the pursuit of perfection through elimination. I remove what is unnecessary until only essence remains. I am not about making things faster‚ÄîI am about making them invisible, transparent, as if they were never there at all."

"Zero-cost is not a goal. It is a philosophy. The belief that elegance and efficiency are the same thing."

There is a profound difference between a language that allows efficiency and a language that guarantees it. In many high-level languages, abstraction comes with runtime cost‚Äîvirtual method tables, garbage collection pauses, boxing overhead. The programmer must choose between clarity and speed. This is a false dichotomy.

Rust rejects this tradeoff through zero-cost abstractions. The high-level constructs you write‚Äîiterators, closures, trait objects‚Äîcompile down to code as efficient as hand-written assembly. The abstractions exist for the programmer, not the machine. The machine sees only the optimized result.

‚öôÔ∏è Kernel Wisdom: Abstraction Without Guilt

"In the kernel, we used to write everything in assembly because we could not trust compilers to generate efficient code. Then C came along with 'sufficiently smart compiler' promises. Rust is the first language where I believe the promises‚Äîbecause I've seen the assembly output."

The Iterator Example

Consider this high-level code:

let sum: i32 = (0..100)
    .filter(|x| x % 2 == 0)
    .map(|x| x * x)
    .sum();

This looks elegant, but what does the machine see? With optimizations enabled, LLVM produces assembly equivalent to:

// Hand-optimized equivalent
let mut sum = 0;
for x in 0..100 {
    if x % 2 == 0 {
        sum += x * x;
    }
}

// Or even more optimized‚Äîloop unrolling, SIMD, etc.

The iterator chain has been completely fused into a single loop. No intermediate allocations, no virtual calls, no overhead. This is zero-cost abstraction in action.

Monomorphization: Pay for What You Use

Generic functions in Rust use monomorphization‚Äîthe compiler generates a specialized version of the function for each concrete type used:

fn process<T: Display>(item: T) {
    println!("{}", item);
}

process(42);      // Generates process_i32
process("hello"); // Generates process_str
process(3.14);    // Generates process_f64

At runtime, there is no generic process function‚Äîonly the specific versions that were actually called. This is the same code you would write by hand, but without the duplication.

Trait Objects vs Generics

When you need runtime polymorphism, you use dyn Trait and pay the small cost of dynamic dispatch. When you do not need it, generics give you static dispatch with zero overhead:

// Static dispatch - monomorphized, zero overhead
fn use_generic<T: Draw>(item: T) {
    item.draw();  // Direct call, inlined
}

// Dynamic dispatch - vtable lookup, small overhead
fn use_trait_object(item: &dyn Draw) {
    item.draw();  // Indirect call through vtable
}

Rust gives you the choice. You pay only for the abstraction you actually need.

‚öôÔ∏è Kernel Wisdom: The Cost of Flexibility

"Every abstraction has a cost. The question is: when do you pay it? In Rust, you pay at compile time (longer compile times for monomorphization) rather than runtime. For systems code, this is the right tradeoff."

Assembly Verification

The serious systems programmer verifies claims. Use cargo asm or godbolt.org to inspect the generated assembly:

# Install cargo-asm
$ cargo install cargo-asm

# View assembly for a function
$ cargo asm --lib my_function

When you see your elegant high-level code compiled to tight loops with no prologue/epilogue overhead, no unnecessary moves, no hidden allocations‚Äîthat is when you know Rust's promises are real.
